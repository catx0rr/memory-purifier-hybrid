# render-rules.md — Deterministic Markdown Rendering

**Locked layout for the five human-facing purified views. `render_views.py` reads `purified-claims.jsonl` + `purified-contradictions.jsonl` + `purified-entities.json` and renders from artifact state. Markdown is derived, never authoritative.**

---

## 1. Core principles

1. **Artifacts are authoritative.** Markdown is always re-derived from the current artifact state. Previous markdown content is overwritten, never appended-to.
2. **Deterministic ordering.** Given the same artifact state, output markdown is byte-identical across reruns. This is what prevents "append-drift" — where repeated runs slowly grow the file with near-duplicate entries.
3. **No raw JSON in markdown.** Users read this. Claims are rendered as prose with minimal formatting; metadata goes into inline italics or backticked labels, not code blocks of JSON.
4. **No prose invention.** Every sentence in the rendered output traces to a `canonical.text` field in the artifacts. The renderer formats and groups; it does not write new prose.
5. **Overwrite safely.** `render_views.py` writes to a temp file first, validates, then atomic-renames over the target. A failed render never leaves a half-written view.

---

## 2. Claim-to-markdown rendering

A single claim renders as ONE block:

```markdown
### <subject> — <predicate> <object-if-present>
<canonical.text>

_Source: <provenance[0].source> L<start>-<end>_
_Status: <status>_  _Confidence: <posture>_  _Updated: <date>_
```

Rules:
- Heading level `###` for individual claims. Home-level headings are `##`. Top-of-file title is `#`.
- If `subject` or `predicate` is empty, omit the dash and fall back to `### <canonical.text truncated to ~80 chars>`.
- Italic metadata line is a single line (even if long) — use ` ` (two spaces) between fields, not `<br>`.
- `Updated` shows `updatedAt` rendered as `YYYY-MM-DD` (strip time).
- Only show `Status` line if status is NOT `canonical`. Canonical claims render without the status field (cleaner).
- If `provenance[]` has more than one entry, render as `_Sources: <s1>, <s2>, …_` (comma-separated).
- `contradictions[]` entries, if present, render as a nested `> Contested by:` blockquote beneath the metadata line.

## 4. PLAYBOOKS.md layout

```markdown
# PLAYBOOKS — Purified Reusable Methods

_Regenerated <YYYY-MM-DD HH:MM Asia/Manila> from `runtime/purified-claims.jsonl`._

---

## <domain-tag-1>

<claims where type ∈ {method, procedure} and secondary_tags includes domain-tag-1, sorted by subject alphabetically>

## <domain-tag-2>

<...>

## Uncategorized

<claims where type ∈ {method, procedure} and no domain tag, sorted by subject alphabetically>
```

Domain tags come from claims' `secondary_tags[]`. If a claim has multiple domain tags, render it under its **first-listed** tag only (never duplicate into multiple sections). Domain-tag sections are sorted alphabetically.

## 5. EPISODES.md layout

```markdown
# EPISODES — Purified Event Digest

_Regenerated <YYYY-MM-DD HH:MM Asia/Manila> from `runtime/purified-claims.jsonl`._
_Full narratives live in `episodes/<slug>.md`. This file is an index._

---

<claims where type == "episode", sorted by provenance[0].captured_at descending (newest first)>
```

Per-episode block:

```markdown
### <YYYY-MM-DD> — <canonical.subject-or-slug>
<canonical.text (digest — 1-3 sentences)>

_Source: [<episodes/<slug>.md>](<relative-path-to-episode-file>) L<start>-<end>_  _Tags: <secondary_tags joined>_
```

Rules:
- `EPISODES.md` claims must have a `source` beginning with `episodes/` per the router's hard constraint 3.3.
- Digest text ≤ 500 chars (validated by `validate_outputs.py`); renderer truncates with `…` if artifact exceeds.
- Link syntax uses relative paths from `<workspace>/`, e.g., `[episodes/retrieval-v1-2-0.md](episodes/retrieval-v1-2-0.md)`.

## 8. Cross-file rules

### 8.1 Regeneration line

Every view starts with:
```markdown
_Regenerated <YYYY-MM-DD HH:MM Asia/Manila> from `runtime/purified-claims.jsonl`._
```

This timestamp uses the local-aware format (not UTC). It's for operator readability, not machine state — the manifest carries the run timestamps.

### 8.2 Do-not-edit notice

The LTMEMORY.md view gets an explicit edit warning:
```markdown
_Do not edit this file by hand — changes will be overwritten on the next purifier run._
```

Other views omit it (less noise); the regeneration line implies it.

### 8.3 Atomic writes

`render_views.py` writes to `<target>.tmp.<pid>`, runs a trailing-newline/UTF-8 check, then `os.rename()` to the final path. If any render step fails, the temp file is unlinked and the previous view remains intact.

### 8.4 Idempotency guarantee

Given the same `purified-claims.jsonl` + `purified-contradictions.jsonl`, the rendered output is byte-identical — except for the regeneration timestamp on line 3. Validators that diff markdown across runs should ignore that one line.

## 9. Anti-patterns — do NOT

- Do NOT append to existing markdown content. Always overwrite from artifact state.
- Do NOT embed raw JSON, code fences of artifact content, or claim metadata beyond the single italic line.
- Do NOT copy `episodes/<slug>.md` body into `EPISODES.md`. Digest only.
- Do NOT create empty sections with headings and no body. Omit sections with zero claims.
- Do NOT duplicate a claim across multiple sections within the same view (use `secondary_tags` first-wins).
- Do NOT write to personal-only views (`HISTORY.md`, `WISHES.md`) on business-profile runs.
- Do NOT produce non-deterministic ordering. Sort keys are explicit for each section.
- Do NOT silently fail a render. Failures write a manifest warning and leave the previous view intact.
